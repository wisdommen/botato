  <action>
Completely rewrite `ai_canvas.gd`. The file extends `Node2D` and is preloaded by `main.gd` (no script extension -- direct preload). Remove ALL existing force calculation logic (the entire current _draw body). Replace with a thin reader that consumes `_last_force_results` from the player's movement behavior node.

**Structure of the new file:**

1. **Constants** at the top of the file:
   - `ARROW_SCALE = 150.0` -- pixels per unit force magnitude
   - `ARROW_MAX_LEN = 80.0` -- maximum arrow length in pixels to prevent visual overflow
   - `ARROW_MIN_LEN = 4.0` -- skip sub-pixel arrows
   - `ARROWHEAD_SIZE = 6.0` -- arrowhead triangle half-width in pixels
   - `SUM_ARROW_WIDTH = 3.0` -- composite sum arrow line width
   - `ENTITY_ARROW_WIDTH = 1.5` -- per-entity arrow line width
   - `BOUNDARY_TYPE_INDEX = 6` -- boundary calculator index in _last_force_results

2. **Color palette** as a class-level array `_type_colors` initialized in `_ready()`. 7 entries indexed to match calculator order in player_movement_behavior.gd. Choose colors optimized for contrast against Brotato's dark arena backgrounds (per D-02). Must leave indices 7+ open for Phase 4 new types. Suggested palette:
   - `[0]` consumable: `Color(0.2, 0.8, 0.2, 0.7)` -- green (health pickups)
   - `[1]` gold: `Color(1.0, 0.84, 0.0, 0.7)` -- gold/yellow
   - `[2]` tree: `Color(0.0, 0.6, 0.4, 0.7)` -- teal (neutrals)
   - `[3]` projectile: `Color(1.0, 0.3, 0.3, 0.7)` -- red (danger)
   - `[4]` enemy: `Color(1.0, 0.5, 0.0, 0.7)` -- orange
   - `[5]` boss: `Color(0.8, 0.0, 0.0, 0.7)` -- dark red (high danger)
   - `[6]` boundary: `Color(0.7, 0.4, 1.0, 0.7)` -- purple (walls)

3. **`_draw_arrow(origin: Vector2, force_vec: Vector2, color: Color, width: float) -> void`** helper:
   - Guard: `if force_vec.length_squared() < 0.0001: return` (prevent NaN from zero vector normalized)
   - Compute direction: `var direction = force_vec.normalized()`
   - Compute length: `var length = min(force_vec.length() * ARROW_SCALE, ARROW_MAX_LEN)`
   - Guard: `if length < ARROW_MIN_LEN: return`
   - Compute tip: `var tip = origin + direction * length`
   - Draw shaft: `draw_line(origin, tip, color, width)`
   - Draw arrowhead as two lines from tip: `var left = tip - (direction.rotated(0.45) * ARROWHEAD_SIZE)` and `var right = tip - (direction.rotated(-0.45) * ARROWHEAD_SIZE)`, then `draw_line(tip, left, color, width)` and `draw_line(tip, right, color, width)`

4. **`_draw() -> void`** main draw method:
   - Wave-end guard: `if $"/root/Main"._wave_timer.time_left < 0.05: return`
   - Visuals enabled guard: `if not $"/root/AutobattlerOptions".enable_ai_visuals: return`
   - Player reference: `var player = $"/root/Main"._players[0]`
   - **Weapon range arc (D-05):** Retain the existing pattern verbatim -- loop `player.current_weapons`, find minimum `weapon.current_stats.max_range`, then `draw_arc(player.position, weapon_range, 0, 6.28, 100, Color.red)`. This is the ONLY standalone computation allowed in _draw -- it is not a force and does not come from _last_force_results.
   - **Movement behavior traversal:** Find the movement behavior child of the player that has `_last_force_results`. Use this approach: iterate `player.get_children()` and find the child node that has the property `_last_force_results` (using `"_last_force_results" in child`). Store reference in a local var `movement_behavior`. If none found, `return` early. This avoids hardcoding the node name (handles assumption A1 safely).
   - Read force results: `var force_results = movement_behavior._last_force_results`. Guard: `if force_results == null or force_results.empty(): return`
   - **Per-entity arrow loop + sum accumulation:**
     ```
     var sum_vector = Vector2.ZERO
     for i in range(force_results.size()):
         var result = force_results[i]
         sum_vector += result.vector
         var color = _type_colors[i] if i < _type_colors.size() else Color.white
         if i == BOUNDARY_TYPE_INDEX:  # boundary -- special origin handling (D-06)
             _draw_boundary_arrows(player, result.debug_items, color)
         else:
             for item in result.debug_items:
                 _draw_arrow(item.position, item.force_vector, color, ENTITY_ARROW_WIDTH)
     ```
   - **Composite sum arrow (D-04):** `_draw_arrow(player.position, sum_vector, Color.white, SUM_ARROW_WIDTH)`

5. **`_draw_boundary_arrows(player, debug_items: Array, color: Color) -> void`** helper for D-06:
   - For each item in debug_items, determine which wall it belongs to by inspecting force_vector direction:
     - `force_vector.y > 0` and `abs(force_vector.y) > abs(force_vector.x)`: top wall -> origin = `Vector2(player.position.x, 0)`
     - `force_vector.y < 0` and `abs(force_vector.y) > abs(force_vector.x)`: bottom wall -> origin = `Vector2(player.position.x, ZoneService.current_zone_max_position.y)`
     - `force_vector.x > 0` and `abs(force_vector.x) > abs(force_vector.y)`: left wall -> origin = `Vector2(0, player.position.y)`
     - `force_vector.x < 0` and `abs(force_vector.x) > abs(force_vector.y)`: right wall -> origin = `Vector2(ZoneService.current_zone_max_position.x, player.position.y)`
   - Then call `_draw_arrow(origin, item.force_vector, color, ENTITY_ARROW_WIDTH)` for each wall arrow.

6. **`_process(_delta) -> void`:** Simply calls `update()` to trigger redraw each frame (identical to current implementation).

**CRITICAL ANTI-PATTERNS -- do NOT include any of these in the rewrite:**
- No `_entity_spawner`, `_consumables`, `_active_golds`, `EnemyProjectiles` references
- No `length_squared()` on entity-to-player vectors (except the zero-guard in `_draw_arrow`)
- No `.normalized()` on entity-to-player vectors (only on force_vec in `_draw_arrow`)
- No weight variables read from AutobattlerOptions (weights are baked into debug_items)
- No `circle_size_multiplier`, `circle_max_size`, or circle rendering (replaced by arrows)
- No `item_weight_squared`, `projectile_weight_squared`, or any squared weight patterns
- No `SpawningAttackBehavior` or entity type checks
